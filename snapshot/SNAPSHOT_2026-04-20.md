## Aktuelles Polling-System (Problem)

### Datei: `bot/observation/sources/solana_polling.py`

**Wie es funktioniert:**
1. Alle X Sekunden: `getSignaturesForAddress` für jede Wallet → Helius RPC
2. Neue Signatures → `getTransaction` → Trade-Event parsen
3. Trade-Event → `wallet_analysis.py` Handler

**Batch-System (aktuell implementiert):**
```python
batch_size = max(3, min(15, len(wallets) // 4 or 3))  # ~4 Batches
RPC_MAX_RPS = 8                                          # konservativ
min_delay   = batch_size / RPC_MAX_RPS
batch_delay = max(batch_delay, min_delay)
```

**Das Problem:**
- Helius Free Tier: **100.000 Credits/Tag**
- Mit 54 Wallets, 5s Poll-Intervall, ~7h/Tag: ~277k Requests/Tag
- Tagesimit wird regelmäßig erschöpft → `max usage reached` Fehler
- Heute auch: `getaddrinfo failed` (DNS-Problem bei Netzwerkausfall)

**RPC-Endpunkt (config/network.py):**
```python
RPC_HTTP_ENDPOINTS = {
    "mainnet": "https://mainnet.helius-rpc.com/?api-key=f607043d-baf5-4bcb-bd7e-c9fca54c5cff"
}
```

## Geplantes Eigenes Polling-System

### Ziel
Kein externer RPC-Provider mehr notwendig. Direkt auf Solana-Netzwerk zugreifen
ohne Credit-Limits.

### Solana RPC Public Endpoints (kostenlos, kein Key)
```
https://api.mainnet-beta.solana.com    (Solana Labs, Hauptendpunkt)
https://solana-api.projectserum.com    (Serum, alternativ)
```
Haben Rate-Limits (~10 req/s) aber keine Credit-Limits.

### Ansatz-Optionen für neues Polling

**Option A: Mehrere Public RPCs rotieren**
- Pool aus mehreren kostenlosen Endpunkten
- Round-Robin oder per Wallet zuweisen
- Kein Credit-Limit, aber Rate-Limit-Management nötig
- Einfachste Implementierung

**Option B: Eigener RPC-Node**
- Solana Validator/Light Node lokal oder auf VPS
- Keine Limits, maximale Kontrolle
- Aufwändig, ~$20-50/Monat für VPS

**Option C: WebSocket statt Polling**
- `logsSubscribe` oder `accountSubscribe` via WebSocket
- Push statt Pull → dramatisch weniger Requests
- Nur neue Transaktionen werden geliefert, kein Polling-Overhead
- Solana Labs Public WS: `wss://api.mainnet-beta.solana.com`

**Empfehlung: Option A + C kombiniert**
- WebSocket für sofortige Trade-Detection (fast keine Requests)
- Polling als Fallback wenn WS abbricht
- Mehrere Public RPCs rotieren für Polling-Fallback

## Wallet-System (aktueller Stand)

### Kategorien in axiom.db
| Kategorie | Anzahl | Bedeutung |
|---|---|---|
| OwnWallet | 1 | Eigene Wallet |
| ActiveWallet | 39 | Laufende Wallets (19 custom + 20 smart) |
| CandidateWallet | 14 | Neue Wallets im Probebetrieb |

### Wallet-Pipeline
```
find_wallets.py (GMGN Browser-Bridge)
      ↓ --apply → max 20 CandidateWallets
axiom_wallets.json
      ↓ python import_wallets.py
axiom.db
      ↓ sync_wallets() → OwnWallet + ActiveWallet + CandidateWallet
Observer beobachtet alle Wallets
      ↓ Session-Ende: _run_session_end_evaluate()
evaluate_wallets.py (EV-Vergleich, mind. 20 Trades)
      ↓ Tausch: Candidate mit höherem EV ersetzt schlechtesten Active
      ↓ find_wallets --apply füllt Candidate-Slots auf
```

### EV-Formel
```
EV = WinRate × AvgWin - (1 - WinRate) × AvgLoss
```
Mindest-Trades für Vergleich: **20 saubere SELLs**

## Wichtige Bugs (behoben)

1. **open_positions Überschreiben** – zwei Wallets kauften denselben Token, zweiter BUY
   überschrieb ersten. Fix: Key von `token` auf `(token, wallet)` geändert.

2. **show_db zeigte nur 14 Wallets** – las nur `wallet_stats`, jetzt LEFT JOIN mit
   `axiom.db` als Basis → alle 53 aktiven Wallets sichtbar.

3. **wallet_stats fehlende Einträge** – Wallets mit nur SESSION_ENDED/CRASH_RECOVERY
   Trades bekamen keinen Eintrag. Fix: Basis-Eintrag bei 0 sauberen Trades.

4. **Lange Sessions (Bug)** – scheinbar 200h Sessions waren Crash-Recovery Trades
   aus Folgesessions die in alte Session-IDs geschrieben wurden. Kein echter Bug,
   Design-bedingt korrekt.
